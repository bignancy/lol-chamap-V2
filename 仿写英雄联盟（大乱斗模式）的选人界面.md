# 仿写英雄联盟（大乱斗模式）的选人界面



### 1. 技术栈选型 (Tech Stack)

- **构建工具：** **Vite** (放弃传统的 Create React App，Vite 启动极快，热更新体验极佳)。
- **开发语言：** **TypeScript** (强烈建议！英雄数据、玩家状态、聊天消息的结构非常复杂，没有 TS 的类型提示，后期极易出 Bug)。
- **样式方案：** **Tailwind CSS** 或 **SCSS (CSS Modules)**。鉴于这类游戏界面高度定制化，如果想写得快，Tailwind 是首选；如果习惯自己手写，用 SCSS 模块化管理避免样式冲突。
- **全局状态管理：** **Zustand**。比 Redux 轻量太多，学习成本低，且极其适合管理“当前房间状态”、“倒计时”、“玩家列表”这种需要在多个组件中共享的数据。
- **网络请求：**
  - **Axios**：用于常规 HTTP 请求（如获取英雄图鉴基础数据、登录获取 Token）。
  - **SockJS + @stomp/stompjs**：用于 WebSocket 实时长连接（如选人动作同步、聊天、倒计时同步）。

### 3. 核心状态数据结构设计 (Zustand Store Design)

这是最重要的一环。你需要思考：页面上有哪些数据是“动”的？把它们抽离出来放到全局 Store 中。

TypeScript

```
// features/ChampSelect/types.ts
export interface Player {
  id: string;
  summonerName: string;
  championId: number | null;
  isLocked: boolean;
  team: 'BLUE' | 'RED';
}

export interface ChampSelectState {
  // 1. 房间基础信息
  roomId: string;
  phase: 'WAITING' | 'PICKING' | 'LOCKED' | 'STARTING'; // 房间所处阶段
  countdown: number; // 倒计时
  
  // 2. 玩家数据
  myTeam: Player[];
  enemyTeam: Player[];
  currentUser: Player; // 当前操作的玩家
  
  // 3. 英雄池数据 (大乱斗顶部的可选英雄池)
  availableChampions: number[]; // 存放英雄 ID 列表
  
  // 4. 聊天数据
  messages: Array<{ sender: string, content: string, type: 'SYSTEM' | 'USER' }>;

  // 5. 操作 Action (供组件调用的方法)
  setCountdown: (time: number) => void;
  updatePlayerPick: (playerId: string, championId: number) => void;
  addChatMessage: (msg: any) => void;
  // ... 其他方法
}
```

### 4. 前后端通信策略 (WebSocket 最佳实践)

选英雄界面的核心是“状态同步”。记住一个原则：**前端只负责展示和发送指令，真正的状态和倒计时由后端 (SpringBoot) 掌控。**

1. **初始化 (HTTP)：** 用户进入页面，发送 HTTP 请求给后端获取当前房间的初始快照（我在哪个队、当前英雄池有谁、倒计时剩多少）。
2. **建立连接 (WebSocket)：** 页面加载完立即建立 WS 连接，订阅当前房间的频道（例如 `/topic/room/12345`）。
3. **用户操作 (前端 -> 后端)：** * 用户点击“掷骰子”（Reroll），前端不直接改数据，而是通过 WS 发送指令 `{"action": "REROLL", "userId": "..."}` 给后端。
   - 用户在聊天框发消息，发送 `{"action": "CHAT", "content": "..."}`。
4. **服务端广播 (后端 -> 前端)：**
   - 后端处理完骰子逻辑，通过 WS 广播给房间所有人：`{"type": "POOL_UPDATE", "newPool": [...]}`。前端 Zustand 监听到这个事件，更新 `availableChampions`，UI 自动重新渲染。
   - **关于倒计时：** 千万不要用前端的 `setInterval` 自己跑倒计时，不同玩家浏览器的性能差异会导致时间不同步。应该由 SpringBoot 后端每秒通过 WS 广播一次 `{"type": "TICK", "time": 6}`，前端只负责把数字显示出来。

### 5. 开发落地步骤建议

为了避免一口吃成个胖子，建议你按以下 3 个阶段来开发：

- **Phase 1：静态切图 (Mock Data)**
  - 不连后端，甚至不用 Zustand。直接在组件里写死假数据（就像你给我的截图中那样）。
  - 目标：把 CSS 样式调好，完美还原深色电竞 UI。
- **Phase 2：引入状态管理 (Local State Flow)**
  - 接入 Zustand 和 TypeScript。
  - 写一些模拟的 Action 函数（比如点击英雄，左侧列表里的头像跟着变）。
  - 目标：跑通前端的单机逻辑，理解数据是如何流向各个组件的。
- **Phase 3：联调后端 (WebSocket Integration)**
  - 引入 `stompjs`。把 Phase 2 里的本地 Action 替换成向后端发消息，并在回调里接收后端的广播更新 Store。
  - 目标：实现双开浏览器，一个号点英雄，另一个号能实时看到变化。
